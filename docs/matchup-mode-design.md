# 対戦モード追加 設計書

## 1. 目的

ダブルス組合せ生成に、以下の対戦モードを追加する。

| モード | 概要 |
| --- | --- |
| 通常 | 現状どおり、同一人物との顔合わせ・同一ペア・同一対戦相手の偏りをなるべく避ける。 |
| 同性対決優先 | 男性同士、女性同士の対戦コートをなるべく優先する。 |
| 混合対決優先 | 男女ペア同士の対戦をなるべく優先する。 |

通常モードは既存挙動を維持する。同性対決優先・混合対決優先は、既存の休憩公平性、連続休憩回避、顔合わせ偏り軽減を壊さず、性別構成の優先度を追加する。

## 2. 前提と優先順位

生成ロジックの優先順位は以下とする。

1. 休憩回数の公平性
2. 連続休憩の回避
3. 対戦モードに合致する性別構成
4. 顔合わせ回数の偏り軽減
5. 同一ペア・同一対戦相手の偏り軽減
6. コート使用回数の均等化

対戦モードは制約ではなく優先条件として扱う。人数構成上、完全には成立しない場合でも生成を失敗させず、既存ロジックへ自然に劣化させる。

## 3. UI 方針

### 3.1 入力項目

条件フォームに `対戦モード` を追加する。

| 項目 | 通常 | 同性対決優先 | 混合対決優先 |
| --- | --- | --- | --- |
| 参加人数 | 入力可 | 入力可 | 入力可 |
| 男性人数 | 入力不可 | 入力必須 | 入力必須 |
| 女性人数 | 入力不可 | 入力必須 | 入力必須 |
| コート数 | 入力可 | 入力可 | 入力可 |
| 実施回数 | 入力可 | 入力可 | 入力可 |

通常モードでは `男性人数` / `女性人数` は入力不可にする。非表示ではなく disabled 表示にすることで、モード切替時に必要になる入力であることを利用者が理解できるようにする。

条件フォーム上には、自動採番の内訳を目立つ形で表示する。同性対決優先・混合対決優先では、女性を先に割り当てる前提が分かるように、例として `01～06：女性、07～14：男性` のような範囲表示にする。

### 3.2 入力検証

同性対決優先・混合対決優先では、以下を検証する。

- `男性人数 + 女性人数 === 参加人数`
- `男性人数 >= 0`
- `女性人数 >= 0`
- 参加人数、コート数、実施回数は既存の最小値制約を維持する

通常モードでは性別人数を生成条件に使わない。通常モードへ戻した場合、性別人数の入力値が残っていても生成結果へ影響させない。

### 3.3 状態管理

数値入力は既存方針と同じく、表示中の入力値がそのまま生成条件になるように扱う。blur 前に生成しても、最新の男性人数・女性人数・対戦モードが反映されることを必須とする。

## 4. データモデル方針

`MatchConditionInput` / `MatchConditions` に対戦モードを追加する。

```ts
type MatchupMode = "standard" | "sameGenderPriority" | "mixedDoublesPriority";
```

`ParticipantInput` / `Participant` に性別を追加する。

```ts
type ParticipantGender = "male" | "female";
```

通常モードでは participant の gender は不要とする。同性対決優先・混合対決優先では、UI の男性人数・女性人数から自動採番参加者へ性別を付与する。

性別人数から自動採番する場合は、女性を先に割り当てる。

例:

- 男性人数 5、女性人数 3 の場合
- `01` から `03` を female
- `04` から `08` を male

将来的に参加者名を個別編集する場合でも、gender は participant 単位に保持できる構造にしておく。

## 5. 生成ロジック方針

### 5.1 通常

通常モードでは性別スコアを常に 0 とし、既存の生成結果を変えない。

### 5.2 同性対決優先

理想は、1コート4人が全員同性で構成されること。

- 4人全員 male のコート
- 4人全員 female のコート

同性4人コートを作れず、結果として1コートが male 2人 / female 2人になった場合だけ、男女ペア同士の対戦に寄せる。それ以外の男女比では、ペア分割は既存の顔合わせ・同一ペア・同一対戦相手回避ロジックへ任せる。

### 5.3 混合対決優先

理想は、1コートが男性2人・女性2人で構成され、かつ男女ペア同士で対戦すること。

評価対象は以下とする。

- 4人グループが male 2人 / female 2人であること
- `pairA` が男女ペアであること
- `pairB` が男女ペアであること

混合対決優先では、4人グループ構成とペア分割の両方に性別スコアを入れる。

## 6. 若いラウンド優先

設定されたモードに合致した対戦は、なるべくラウンドの若い方に組まれることを理想とする。

このため、性別条件の不一致ペナルティにはラウンド重みを持たせる。

例:

```ts
const roundWeight = roundCount - roundIndex;
genderPreferencePenalty += mismatchPenalty * roundWeight;
```

同じ達成数の候補がある場合、序盤ラウンドでモード条件を満たす seed が選ばれやすくなる。後半ラウンドの品質を捨てるわけではなく、seed 探索時の比較で序盤一致を少し優先する。

## 7. スコア設計

`ResultScore` に `genderPreferencePenalty` を追加する。

```ts
type ResultScore = {
  fairnessPenalty: number;
  consecutiveRestPenalty: number;
  genderPreferencePenalty: number;
  encounterPenalty: number;
  sameTeammatePenalty: number;
  sameOpponentPenalty: number;
  totalScore: number;
};
```

`totalScore` は既存の重みに加えて、対戦モード用の重みを追加する。

通常モードでは `genderPreferencePenalty = 0` とし、既存の score 比較へ影響させない。

`generateMatchupUseCase` の候補 seed 比較では、`totalScore` の次、または `encounterPenalty` より前に `genderPreferencePenalty` を比較対象へ入れる。性別モードを選んだ場合は、顔合わせ偏りよりもモード一致を優先するため。

## 8. 共有URL・再現性

対戦モードと性別人数は、同一 seed 再現に必要な生成条件である。そのため、共有URLには以下を含める。

- `mode`
- `male`
- `female`
- `participants`
- `courts`
- `rounds`
- `seed`

共有URLスキーマの version は更新する。旧 version の互換を残す場合は、旧URLを通常モードとして復元する。互換を切る場合は、version 境界として明示する。

replay では `generateMatchupUseCase` ではなく、復元した条件と採用 seed を使って `generateMatchup()` を直接呼ぶ。seed 探索を再実行すると、共有元と異なる結果になる可能性があるため。

## 9. API 影響

`POST /api/v1/matchups/generate` と `POST /api/v1/matchups/replay` の入力に以下を追加する。

- `matchupMode?: "standard" | "sameGenderPriority" | "mixedDoublesPriority"`
- `participants[].gender?: "male" | "female"`

API では後方互換のため、`matchupMode` 未指定を `standard` として扱う。`sameGenderPriority` / `mixedDoublesPriority` の場合は、全 participant に gender があることを必須にする。

UI の `男性人数` / `女性人数` は API の直接項目にはせず、UI 側で participant 配列へ変換する。API 利用者は参加者ごとに gender を指定できるため、将来的な名前付き参加者にも対応しやすい。

## 10. 印刷・PDF・表示

結果表示、印刷、PDF では、同性対決優先・混合対決優先の場合に参加者の性別を識別できる表示を追加する。

表示は参加者番号の横に半角の性別マークを付ける。対戦表上の幅を抑えるため、全角文字やサイズ差の出やすい記号は使わない。

| 性別 | 表示 |
| --- | --- |
| female | `F` |
| male | `M` |

例:

- `01F / 09M`

候補:

- 参加者一覧・休憩者一覧で性別を併記
- コートカード内のペア表示に性別記号を併記

通常モードでは性別表示を出さない。

## 11. テスト設計

### 11.1 テスト観点

機能観点:

- 通常モードで既存ロジックと同じ条件生成になること
- 同性対決優先で4人全員同性のコートが優先されること
- 同性対決優先で male 2人 / female 2人のコートになった場合だけ、男女ペア同士が優先されること
- 混合対決優先で male 2人 / female 2人、男女ペア同士が優先されること
- 若いラウンドでモード一致が優先されること

非機能観点:

- seed が同じなら同じ結果を再現できること
- 候補 seed 探索の実行時間が大きく悪化しないこと
- 人数構成上、理想条件が成立しない場合でも生成が完了すること

データ観点:

- 男性人数 + 女性人数 = 参加人数
- 男性人数 + 女性人数 != 参加人数
- 男性0人、女性0人
- 男性1人、女性多数
- 男性2人、女性2人
- 参加人数がコート数に対して不足するケース

UI観点:

- 通常モードでは男性人数・女性人数が入力不可になること
- ②③では男性人数・女性人数が入力可能かつ必須になること
- ②③では女性から先に番号が割り当てられること
- ②③ではフォーム上に `01～06：女性、07～14：男性` のような採番内訳が表示されること
- ②③では対戦表上の番号横に `F` / `M` が表示されること
- モード切替後、表示中の値で生成されること
- blur 前に生成しても最新入力値が使われること
- エラーメッセージが入力項目と対応していること

### 11.2 正常系

- 通常モード、8人、2面、4回で既存テストが通ること
- 同性対決優先、男性8人、女性0人で全コートが同性構成になること
- 同性対決優先、男性4人、女性4人、2面で同性コートが優先されること
- 同性対決優先、男性2人、女性2人、1面で男女ペア同士が優先されること
- 混合対決優先、男性4人、女性4人、2面で各コートが2M2Fかつ男女ペアになること

### 11.3 異常系

- ②③で男性人数 + 女性人数が参加人数と一致しない場合、生成しないこと
- ②③で gender が不足した API request を拒否すること
- 共有URLの mode / male / female が不正な場合、復元エラーになること

### 11.4 境界値

- 参加人数4人、1面、1回
- 男性0人 / 女性4人
- 男性4人 / 女性0人
- 男性1人 / 女性3人
- 男性2人 / 女性2人
- 参加人数が4で割り切れず休憩者が出るケース

### 11.5 状態遷移

- 通常から同性対決優先へ変更し、性別人数を入力して生成
- 同性対決優先から混合対決優先へ変更し、同じ性別人数で生成
- ②③から通常へ戻した場合、性別人数が結果に影響しないこと
- 共有URL復元後、再作成しても同じ mode / gender 条件が使われること

## 12. 実装ステップ

1. 型・schema・条件構築へ `matchupMode` / `gender` を追加する。
2. UI に対戦モード、男性人数、女性人数を追加する。
3. 自動参加者生成で性別を付与する。
4. 4人グループ選定に性別構成スコアを追加する。
5. 混合対決優先と、同性対決優先で 2F2M になった場合のペア分割に男女ペアスコアを追加する。
6. `ResultScore` と seed 比較へ `genderPreferencePenalty` を追加する。
7. 共有URL version と復元処理を更新する。
8. API schema と設計書を更新する。
9. 結果表示、印刷、PDFの性別表示方針を反映する。
10. 単体テスト、schema テスト、共有復元テスト、UI/E2E テストを追加する。

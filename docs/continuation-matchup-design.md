# 途中参加・退出時の残りラウンド再作成 設計書

## 1. 目的

対戦表作成後、途中で参加者が増える、または減る場合に、実施済みラウンドを固定したまま、現ラウンド以降の対戦表だけを再作成できるようにする。

既存の全体再作成は、全ラウンドを新しい条件で作り直す機能として維持する。本機能は、進行中のイベントで実施済み結果を壊さずに、以降のラウンドへ参加者増減を反映するための追加機能とする。

本機能はアプリバージョン `1.2.0` での追加を予定する。

## 2. 前提

- 現行の参加者は、自動採番された `01`, `02`, ... の番号参加者として扱う。
- 実施済みラウンドは、先頭から連続した範囲のみ有効とする。
- 実施済みラウンドの対戦内容は変更しない。
- 途中変更は「現ラウンド開始前」に行う操作とする。
- 途中再作成後も、既存の `MatchupResult` で表現できる範囲で画面表示、印刷、PDF 出力を行う。
- 共有 URL と API の新スキーマ対応は今回の対象外とする。

## 3. 用語定義

| 用語 | 定義 |
| --- | --- |
| 実施済み連続ラウンド | Round 1 から連続して実施済みになっているラウンド数。 |
| 現ラウンド | `実施済み連続ラウンド + 1`。途中変更の対象開始ラウンド。 |
| 固定ラウンド | 実施済み連続ラウンドに含まれるラウンド。途中再作成で内容を変更しない。 |
| 再作成対象ラウンド | 現ラウンド以降の未実施ラウンド。 |
| 全参加者履歴 | 途中退出者、途中参加者を含む、対戦表全体の表示に必要な参加者一覧。 |
| 今後の生成対象者 | 現ラウンド以降の対戦表生成に参加できる参加者一覧。退出者は含めない。 |
| 途中再作成結果 | 固定ラウンドと、途中変更後に再生成した未実施ラウンドを結合した結果。 |

## 4. 対象範囲

### 4.1 対象

- 実施済みラウンドの固定
- 現ラウンド以降の再作成
- 途中退出者の番号指定
- 途中参加者の追加
- 途中参加者を履歴 0 として扱うこと
- 途中再作成の複数回実施
- 通常の組合せ作成／全体再作成時、参加人数に対してコート数が過多な場合の確認と自動調整
- 途中再作成時、再生成対象ラウンドの使用コートを若い番号へ詰めること
- 参加者スタッツの折りたたみ
- 途中再作成後の組合せ共有ボタン無効化
- 既存 `MatchupResult` で表現できる範囲の印刷、PDF 出力

### 4.2 対象外

- 共有 URL の新スキーマ対応
- API の途中再作成対応
- 途中退出者、途中参加者の状態ラベル表示
- 退出で空いた番号の再利用
- 実施済みラウンドの任意位置変更
- 現ラウンド開始後の途中変更
- 全参加者履歴が既存上限を超えるケースの緩和

## 5. ラウンド進行仕様

### 5.1 現ラウンド

現ラウンドは以下で定義する。

```txt
現ラウンド = 実施済み連続ラウンド + 1
```

例:

| 実施済み連続ラウンド | 現ラウンド | 再作成対象 |
| --- | --- | --- |
| 0 | Round 1 | Round 1 以降 |
| 2 | Round 3 | Round 3 以降 |
| 5 / 全5回 | なし | なし |

### 5.2 実施済み変更

実施済み状態は、先頭から連続した範囲だけを許可する。

- 現ラウンドだけ、未実施から実施済みに変更できる。
- 直前の実施済みラウンドだけ、実施済みから未実施に戻せる。
- 途中再作成で固定ラウンドとして使った実施済みラウンドは、以降は未実施に戻せない。
- それ以外のラウンドは、実施済み/未実施を変更できない。

例: Round 1, Round 2 が実施済みの場合:

| ラウンド | 操作 |
| --- | --- |
| Round 1 | 変更不可 |
| Round 2 | 未実施へ戻せる |
| Round 3 | 実施済みにできる |
| Round 4 以降 | 変更不可 |

実装上は、実施済み状態を `number[]` ではなく `completedRoundCount` として扱う方が、非連続状態を防ぎやすい。

途中再作成後は、再作成時点の `completedRoundCount` を `lockedCompletedRoundCount` として保持する。ロック済みの実施済みラウンドは、残りラウンド生成の履歴前提になっているため、直前ラウンドであっても未実施へ戻せない。途中再作成後に新たに完了にした現ラウンドは、次の途中再作成でロックされるまでは直前戻し可能とする。

## 6. 参加者増減仕様

### 6.1 退出者

- 退出者は、今後の生成対象者から除外する。
- 退出者は、過去ラウンド表示のため全参加者履歴には残す。
- 退出者が現ラウンドで出場予定でも休憩予定でも、現ラウンド以降の対象から除外する。
- 退出済み参加者は、次回以降の途中再作成でも今後の生成対象者へ戻さない。

### 6.2 途中参加者

- 途中参加者は、全参加者履歴へ追加する。
- 途中参加者は、今後の生成対象者へ追加する。
- 途中参加者は、過去ラウンドの出場、休憩、顔合わせ履歴を 0 として扱う。
- 採番は、全参加者履歴の最大番号 + 1 から行う。
- 退出で空いた番号は再利用しない。

例:

```txt
初期参加者: 01〜08
03 が退出
1人追加
=> 追加者は 09
```

### 6.3 性別モード

通常モードでは、追加人数だけを指定する。

性別優先モードでは、以下を指定する。

- 追加女性数
- 追加男性数

追加者の採番順は、既存の自動採番方針と同じく女性を先に割り当て、その後に男性を割り当てる。

例:

```txt
既存最大番号: 08
追加女性数: 1
追加男性数: 2
=> 09F, 10M, 11M
```

### 6.4 制約

- 今後の生成対象者が 4 人未満になる場合、途中再作成はできない。
- 全参加者履歴は、既存の参加人数上限を超えない範囲で扱う。
- 追加後の合計ラウンド数は、既存のラウンド数上限を超えない範囲で扱う。
- 途中再作成成功後、追加ラウンド数、退出者選択、追加人数入力はリセットする。

## 7. 途中再作成仕様

### 7.1 基本動作

途中再作成では、以下を行う。

1. 実施済み連続ラウンドを固定ラウンドとして取り出す。
2. 実施済みラウンドを生成履歴へ反映する。
3. 退出者を今後の生成対象者から除外する。
4. 途中参加者を全参加者履歴と今後の生成対象者へ追加する。
5. 追加ラウンド数が指定されている場合、合計ラウンド数を増やす。
6. 現ラウンド以降だけを再生成する。
7. 固定ラウンドと再生成ラウンドを結合して `MatchupResult` を更新する。

全ラウンド実施済みの場合でも、追加ラウンド数が 1 以上であれば、次のラウンドから新規生成できる。

実施済みラウンドの履歴は、既存の `updateStats()` と同等の処理で、出場回数、休憩回数、顔合わせ、ペア、対戦相手、コート使用履歴へ反映する。

### 7.2 複数回実施

途中再作成は複数回実施できる。

各回の再作成対象は、その時点の現ラウンド以降とする。

例:

```txt
初期: 01〜08 / Round 1〜5

Round 1〜2 実施済み
=> 03 退出、09 追加
=> Round 3〜5 再作成

Round 3 実施済み
=> 05 退出、10 追加
=> Round 4〜5 再作成
```

この場合、Round 1〜3 は固定され、Round 4〜5 だけが再生成される。`03` と `05` は今後の生成対象から除外され、`09` と `10` は今後の生成対象に含まれる。

### 7.3 既存全体再作成との関係

既存の `組合せ作成／再作成` ボタンは、全体再作成として扱う。

- 全体再作成では、途中再作成状態を維持しない。
- 全体再作成では、実施済みラウンドの固定を行わない。
- 途中再作成は、途中変更パネルの専用ボタンでのみ実行する。

### 7.4 コート数過多時の扱い

通常の `組合せ作成／再作成` では、参加人数に対して入力コート数が多い場合、生成前に確認ダイアログを表示し、OK 時だけコート数を自動調整してから生成する。

判定条件:

```txt
4 <= 参加人数 < コート数 × 4
```

調整後コート数:

```txt
Math.floor(参加人数 / 4)
```

例:

| 参加人数 | 入力コート数 | 調整後コート数 |
| --- | --- | --- |
| 4〜7 | 2 | 1 |
| 8〜11 | 3 | 2 |
| 12〜15 | 4 | 3 |
| 30 | 8 | 7 |

`キャンセル` の場合は、生成せず、入力値も変更しない。`OK` の場合は、画面上のコート数入力も調整後の値へ更新し、調整済み input で生成する。

途中変更パネルの `Round N 以降を再作成` では、確認ダイアログを表示せず、コート数も減らさない。固定済みラウンドとの表示整合性を保つため、`conditions.courtCount` は維持する。ただし、再生成される未来ラウンドだけは、使用コートを若い番号へ詰める。

例:

```txt
7人 / 2面 / Round 1〜2 実施済み
=> Round 3 以降を途中再作成
=> Round 3 以降は コート1 を使用、コート2 は未使用
```

## 8. 画面表示仕様

### 8.1 表示順

途中変更パネルは、既存の対戦表の下に表示する。

```txt
条件入力フォーム
↓
結果サマリー
↓
対戦表 Round 1
対戦表 Round 2
対戦表 Round 3
...
↓
途中変更パネル
↓
参加者別サマリー 見出し
↓
印刷 / PDF などの操作
```

途中変更は、現在の対戦表を確認した後に行う補助操作である。そのため、条件入力フォーム側ではなく結果エリア側に配置する。

作成、全体再作成、途中再作成が成功した場合は、画面位置に依存せず完了を認識できるように、数秒で自動消去される案内メッセージ `作成/再作成が完了しました。` を表示する。

### 8.2 コート数調整確認ダイアログ

通常の `組合せ作成／再作成` ボタン押下時に、コート数過多の条件を満たす場合は確認ダイアログを表示する。

表示文言:

```txt
確認

コート数を調整します

入力されたコート数 2面 は参加人数に対して多いため、
コート数：1面で対戦表を作成します。

[キャンセル] [OK]
```

表示値は入力値と調整後コート数に応じて差し替える。

- `キャンセル`: ダイアログを閉じ、作成しない。入力値も変更しない。
- `OK`: ダイアログを閉じ、コート数入力を調整後へ更新し、調整済み条件で作成/再作成する。
- 生成後は、通常の完了メッセージ `作成/再作成が完了しました。` を表示する。

OK 時の生成は、React state の反映待ちに依存しない。調整済みの `MatchConditionInput` を直接作成し、その input を `generate` / `regenerate` に渡す。

途中変更パネルの `Round N 以降を再作成` では、この確認ダイアログは表示しない。

### 8.3 参加者別サマリー

参加者別サマリーは初期状態で非表示にする。

- 見出し行に開閉ボタンを表示する。
- 閉じている場合は `ChevronDown` 相当の表示にする。
- 開いている場合は `ChevronUp` 相当の表示にする。
- 開閉状態は画面内 state で管理する。
- 開いている場合だけ、同じカード内に `PlayerStatsTable` を表示する。
- 展開時に追加の見出しは表示しない。

表示例:

```txt
参加者別サマリー [▼]
```

開いた状態:

```txt
参加者別サマリー [▲]

参加者 | 出場 | 休憩 | ...
...
```

### 8.4 途中変更パネル

途中変更パネルは初期状態で非表示にする。

- 見出し行に開閉ボタンを表示する。
- 閉じている場合は `ChevronDown` 相当の表示にする。
- 開いている場合は `ChevronUp` 相当の表示にする。
- 開閉状態は画面内 state で管理する。
- 開いている場合だけ途中変更フォームを表示する。

途中変更パネルには、以下を表示する。

- 追加ラウンド数
- 追加する参加者
- 退出者の番号
- 変更後の参加人数
- 途中再作成ボタン

画面イメージ:

```txt
ラウンド・参加者途中変更 [▼]
```

開いた状態:

```txt
ラウンド・参加者途中変更 [▲]

⚠ 共有リンクは再生成後に無効になります。

▼ 詳細
・事前に終了ラウンドは実施済み☑に変更
・人数変更は未実施のラウンドにのみ反映
・初回作成時の対戦モード・人数を基準に再作成

追加ラウンド数 [-] 0 [+]

追加する参加者
追加人数 [-] 1 [+]

退出者の番号
[01] [02] [03] [04] [05] [06] [07] [08]
選択中: 03

変更後の参加人数：8人 → 8人
[Round 3以降を再作成]
```

`退出者の番号` は参加者履歴を表示対象にする。すでに退出済みで今後の生成対象から外れている番号は、無効状態で表示し、再選択できないようにする。

性別優先モードの場合:

```txt
追加する参加者
追加女性 [-] 0 [+] / 追加男性 [-] 1 [+]
```

### 8.5 退出者 UI

退出者指定は番号チップ選択とする。

自由入力欄ではなく番号チップにする理由:

- 存在しない番号の入力を防げる。
- 全角、半角、区切り文字違いを考慮しなくてよい。
- 重複指定を UI 上で防ぎやすい。
- 最大 30 人のため、チップ一覧でも表示可能。

退出者チップは、今後の生成対象者だけを選択可能にする。既に退出済みの参加者は選択対象に出さない。

### 8.6 ボタン状態

途中再作成ボタンは、以下の場合に無効化する。

- 結果が存在しない。
- 現ラウンドが存在しない。
- 参加者変更がない。
- 変更後の今後の生成対象者が 4 人未満。
- 全参加者履歴が既存上限を超える。
- 性別優先モードで追加女性数、追加男性数の入力が不正。

## 9. データモデル・状態管理方針

### 9.1 UI 状態

画面側では、以下の状態を持つ。

```ts
type ResultSource = "full" | "continuation";

type ContinuationDraft = {
  withdrawnParticipantIds: string[];
  addCount: number;
  addFemaleCount: number;
  addMaleCount: number;
};
```

`ResultSource` は、途中再作成後の組合せ共有ボタン無効化に使う。API レスポンス契約へ影響させないため、`MatchupResult` へ直接フラグを足すより、画面または store 側の状態として持つ。

### 9.2 実施済み状態

実施済み状態は、以下のように連続数で管理する。

```ts
type CompletedRoundsState = {
  generatedAt: string | null;
  completedRoundCount: number;
  lockedCompletedRoundCount: number;
};
```

`completedRoundCount` にすることで、Round 1 と Round 3 だけが実施済みになるような非連続状態を防ぐ。

`lockedCompletedRoundCount` は、途中再作成で固定ラウンドとして使った実施済み範囲を表す。これにより、再作成後に固定済みラウンドを未実施へ戻して、生成前提と画面状態がずれることを防ぐ。

### 9.3 全参加者履歴と今後の生成対象者

途中再作成では、参加者を以下に分ける。

```ts
type ContinuationParticipants = {
  allParticipants: Participant[];
  eligibleParticipantIds: string[];
};
```

- `allParticipants` は表示、印刷、PDF、過去ラウンド参照に使う。
- `eligibleParticipantIds` は現ラウンド以降の生成候補に使う。

既存の `conditions.participants` は表示用の全参加者履歴として扱う。生成ロジックでは、今後の生成対象者だけを休憩者選定とコート割当に使えるようにする。

## 10. ドメイン実装方針

### 10.1 新規ユースケース

既存の `generateMatchup()` は全体生成として残す。

途中再作成用に、以下のような新規ユースケースを追加する。

```ts
generateContinuationMatchupUseCase(input, seed): MatchupResult
```

責務:

- 固定ラウンドの抽出
- 参加者増減の反映
- 生成履歴の初期化
- 現ラウンド以降の再生成
- 途中再作成用の compact コート割当指定
- 固定ラウンドと再生成ラウンドの結合
- 候補 seed の比較

### 10.2 生成コンテキスト

`GenerationContext` は、全参加者履歴と今後の生成対象者を区別できるように拡張する。

```ts
type GenerationContext = {
  conditions: MatchConditions;
  eligibleParticipantIds?: string[];
  // existing fields...
};
```

`eligibleParticipantIds` がない場合は、従来どおり `conditions.participants` 全員を生成対象とする。これにより、既存の全体生成への影響を最小化する。

### 10.3 履歴反映

途中再作成では、固定ラウンドを先に `updateStats()` 相当で反映する。

反映対象:

- 出場回数
- 休憩回数
- コート別出場回数
- コート使用回数
- 顔合わせ回数
- 同一ペア回数
- 同一対戦相手回数
- 休憩履歴
- 出場履歴

その後、現ラウンド以降を生成する。

### 10.4 評価対象

候補 seed の比較では、今後の生成対象者を主な評価対象とする。

- 退出者は、今後の休憩公平性や出場公平性の評価対象にしない。
- 途中参加者は、履歴 0 として今後の公平性評価に含める。
- 固定ラウンドで発生した顔合わせ、ペア、対戦相手の履歴は、残りラウンドの重複回避に使う。

結果表示用の stats は、全参加者履歴に対して算出する。これにより、過去ラウンドに登場した退出者も既存の表示形式で扱える。

### 10.5 コート割当モード

通常の全体生成では、既存どおりコート使用回数の偏りを減らすため、使用コート番号を分散する。

途中再作成では、固定済みラウンドとの `conditions.courtCount` は維持しつつ、現ラウンド以降の再生成ラウンドだけ使用コートを若い番号へ詰める。これは、途中再作成後の画面確認で「使うコート」と「未使用コート」を直感的に把握しやすくするためである。

実装上は、`generateMatchup()` にコート割当モードを渡せるようにする。

```ts
type CourtAssignmentMode = "balanced" | "compact";

generateMatchup(conditions, seed, {
  initialRounds,
  eligibleParticipantIds,
  courtAssignmentMode: "compact",
});
```

- `balanced`: 既存の通常作成、全体再作成、共有復元で使う。コート使用回数を分散する。
- `compact`: 途中再作成で使う。active court を `1..使用面数` に固定し、残りを未使用コートにする。

`compact` はコート番号の割当にだけ影響させる。休憩者選定、ペアリング、対戦相手の重複回避は既存ロジックを維持する。

## 11. 共有・印刷・PDF

### 11.1 組合せ共有

途中再作成結果では、組合せ共有ボタンを無効化する。

理由:

- 既存の共有 URL は、同一条件と seed から全ラウンドを再現する前提である。
- 途中参加、途中退出、固定ラウンドを含む結果は、既存スキーマでは表現できない。
- 新スキーマ対応は今回の対象外である。

通常の新規作成、全体再作成結果では、従来どおり組合せ共有を利用できる。

### 11.2 印刷・PDF

印刷と PDF は、既存 `MatchupResult` で表現できる範囲で対応する。

- 固定ラウンドと再生成ラウンドを結合した対戦表を出力する。
- 退出者、途中参加者の状態ラベルは表示しない。
- 途中再作成結果では、印刷と PDF の組合せ共有 QR を表示しない。
- 途中再作成結果であることを PDF スキーマとして追加表現することは今回対象外とする。

## 12. エラー・バリデーション

| 区分 | 条件 | 表示方針 |
| --- | --- | --- |
| 変更なし | 追加ラウンド、退出者、追加者がいずれもない | 途中再作成ボタンを無効化 |
| 人数不足 | 変更後の今後の生成対象者が 4 人未満 | エラーメッセージ表示 |
| 上限超過 | 全参加者履歴が既存上限を超える | エラーメッセージ表示 |
| ラウンド上限超過 | 追加後の合計ラウンド数が既存上限を超える | エラーメッセージ表示 |
| コート数過多 | 通常作成/全体再作成で `4 <= 参加人数 < コート数 × 4` | 確認ダイアログを表示し、OK 時だけ調整後コート数で生成 |
| 現ラウンドなし | 全ラウンド実施済みかつ追加ラウンド数が 0 | 途中再作成不可 |
| 性別入力不正 | 性別優先モードで追加人数入力が不正 | エラーメッセージ表示 |
| 内部生成失敗 | 生成ロジックで例外 | 既存エラー表示に準拠 |

## 13. テスト設計

### 13.1 テスト観点

機能観点:

- 実施済みラウンドが固定されること。
- 現ラウンド以降だけが再生成されること。
- 途中退出者が以降に出場しないこと。
- 途中参加者が以降に出場できること。
- 追加ラウンド数に応じて合計ラウンド数が増えること。
- 全ラウンド実施済みでも、追加ラウンドがあれば次ラウンドから生成できること。
- 途中再作成を複数回実施できること。
- 既存全体再作成は従来どおり全体を再生成すること。
- 通常作成/全体再作成では、参加人数に対してコート数が過多な場合に確認後、調整後コート数で生成されること。
- 途中再作成では、コート数を減らさず、再生成ラウンドの使用コートだけ若い番号へ詰まること。

データ観点:

- 全参加者履歴と今後の生成対象者が分離されること。
- 途中参加者の履歴が 0 として扱われること。
- 退出者の過去ラウンド表示が維持されること。
- 採番が最大番号 + 1 で継続すること。
- 性別優先モードで追加者の gender が正しく付与されること。
- コート数調整後の `conditions.courtCount` が調整後コート数になること。
- 途中再作成結果では、固定ラウンドと再生成ラウンドで同じ `conditions.courtCount` を維持すること。

UI 観点:

- 現ラウンドと直前実施済みラウンドだけ操作できること。
- コート数調整確認ダイアログの OK/キャンセルが期待どおり動くこと。
- 退出者チップが選択可能状態と選択不可状態を正しく表示すること。
- 参加者スタッツが初期非表示で、開閉できること。
- 途中再作成後に組合せ共有ボタンが無効化されること。
- 30 人表示時でもチップ一覧がスマートフォン幅で崩れないこと。

非機能観点:

- 途中再作成時に過度な待ち時間が発生しないこと。
- 再作成ボタン連打で状態が二重適用されないこと。
- コート数調整確認中に二重生成されないこと。
- 既存の新規作成、全体再作成、共有復元に影響しないこと。
- 印刷、PDF 出力が既存 `MatchupResult` の範囲で破綻しないこと。

### 13.6 通し実行で注意する点

- 途中再作成の前後で、固定ラウンドの内容を比較できる証跡を残す。
- 退出者が以降の `activePlayerIds` と `restPlayerIds` に含まれないことを確認する。
- 追加者が固定ラウンドに含まれず、現ラウンド以降だけに登場することを確認する。
- 単体では成功しても、複数回途中再作成で採番や退出状態が崩れないことを確認する。
- E2E では、再作成操作を並列実行せず、同一ブラウザコンテキストで順番に検証する。
